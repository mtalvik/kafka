# Транзакция на одном банане

Kafka сама по себе уже даёт partition, offset, идемпотентность. Транзакция —
это **ещё один слой сверху**, и честный вопрос: зачем? Разберём на самом
важном бизнесе в мире — продаже одного банана.

---

## Сцена

Онлайн-магазин, один сервис `checkout`. Клиент купил **1 банан**. Заказ
прилетает во входной топик `orders`. Чтобы обработать этот один заказ,
сервис делает **три записи в Kafka**:

1. в топик `inventory` → «склад: −1 банан»
2. в топик `receipts` → «чек: 1 банан, 50 центов»
3. отметить offset заказа как «обработан» (commit)

Запомни: **обработка одного банана = три записи.** Вот тут и живёт
транзакция.

---

## Без транзакции — где ломается

Три записи независимы. Сервис падает **после 1 и 2, но до 3**:

```
читает заказ "1 банан"
  ├─ inventory: -1 банан   ✓ записал
  ├─ receipts: чек         ✓ записал
  ✗ ПАДЕНИЕ (offset НЕ закоммичен)
  │
рестарт → читает ТОТ ЖЕ заказ снова (offset не сдвинулся)
  ├─ inventory: -1 банан   ✓ ЕЩЁ РАЗ
  ├─ receipts: чек         ✓ ЕЩЁ РАЗ
  └─ offset: commit        ✓
```

Клиент купил **1** банан. Склад ушёл в **−2**. Чека **два**. Два банана
из воздуха.

Обратный случай: закоммитил offset, упал до чека → деньги взяли, чека
нет, банан не отгрузили. Тоже плохо.

**Корень:** три записи должны примениться **все или ни одной**. По
отдельности — нельзя.

---

## С транзакцией — где чинится

```
producer.beginTransaction();          ◄── ВОТ ГДЕ НАЧИНАЕТСЯ
    send(inventory, "-1 банан")
    send(receipts, "чек 50¢")
    sendOffsetsToTransaction(offset заказа)   ← коммит offset ВНУТРИ транзакции
producer.commitTransaction();         ◄── ВОТ ГДЕ КОНЧАЕТСЯ (успех)
```

- **Начало** — `beginTransaction()`: «я открываю пачку записей, которые
  применятся вместе».
- **Внутри** — все три записи (склад, чек, offset). Пока транзакция
  открыта, читатели с `read_committed` их **не видят**.
- **Конец** — `commitTransaction()`: все три становятся видимыми **в один
  момент**. Либо `abortTransaction()` / падение: **ни одна** не видна,
  offset не сдвинулся.

Теперь падение посреди:

```
begin
  ├─ inventory: -1   (записал, но НЕ видно)
  ├─ receipts: чек   (записал, но НЕ видно)
  ✗ ПАДЕНИЕ до commit
  │
Kafka сама абортит транзакцию → все три записи выброшены, offset на месте
  │
рестарт → читает заказ заново, с чистого листа
begin → inventory -1 → чек → offset → commit ✓
```

Ровно **один** банан продан. Один раз списан, один чек.

---

## Суть в одной мысли

Это **та же самая транзакция, что в базе данных**:

```
BEGIN
   ... несколько операций ...
COMMIT      -- всё или ничего
```

Только у Kafka «операция» = сообщение в топик. И хитрость: **commit
offset — это тоже запись в топик** (`__consumer_offsets`). Поэтому его
можно засунуть внутрь транзакции вместе со складом и чеком — и тогда
«отметить заказ обработанным» и «списать банан» становятся
**неделимыми**.

- **Где начинается:** `beginTransaction()`.
- **Где кончается:** `commitTransaction()` (успех) или `abortTransaction()`
  / падение (откат).
- **Что даёт:** между этими двумя точками — либо все записи видны, либо
  ни одной. Никаких «полутора бананов».

---

Транзакция — не «ещё одна абстракция ради абстракции». Это просто
**скобки** `begin…commit` вокруг записей, которые обязаны жить или умереть
вместе. Без них Kafka гарантирует каждую запись по отдельности — но не
то, что они согласованы между собой.

> Связь с кодом лабы: этот банан — ровно `Ex6ReadWrite`. Там
> `orders → преобразование → outbound` плюс `sendOffsetsToTransaction`
> в одной транзакции. Замени «преобразование» на «списать склад + чек» —
> и получится этот сценарий один-в-один.
