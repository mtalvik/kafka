# Идемпотентный producer на одном банане

Продолжаем с тем же магазином. В файле про транзакции мы чинили дубли,
которые возникают, когда сервис **падает** посреди работы. Здесь — другой
дубль, который рождается вообще без падений, просто из-за сети. Его лечит
не транзакция, а идемпотентный producer. Это два разных дубля из двух
разных мест — и половина путаницы в том, что их смешивают.

---

## Сцена

Клиент нажал «Купить банан» **один раз**. Сервис `checkout` отправляет
одну запись в топик `orders`: «заказ: 1 банан». Больше клиент ничего не
делал — один клик, одно сообщение.

Между сервисом и брокером — сеть. Сеть иногда врёт.

---

## Где рождается дубль (без всякого падения)

Producer отправляет запись, брокер её записывает и шлёт назад
подтверждение — `ack`. И вот `ack` теряется по дороге:

```
checkout                          Broker
   |                                |
   |  send("1 банан")  ------------>|  [записал ✓]  orders: [банан]
   |                                |
   |  <---- ack (ПОТЕРЯЛСЯ ✗) ------|
   |                                |
   |  таймаут... "не дошло?"        |
   |  retry: send("1 банан") ------>|  [записал СНОВА]  orders: [банан, банан]
   |                                |
   |  <-------- ack ----------------|
```

Ключевой момент: **брокер записал заказ нормально с первого раза.**
Проблема только в том, что producer не узнал об этом — `ack` пропал. С
точки зрения producer'а: «я отправил, ответа нет, значит надо повторить».
Он повторяет. Брокер видит вторую запись и не может понять, что это тот же
заказ — записывает.

Клиент кликнул **один раз**, а в `orders` лежат **два банана**. Дальше по
пайплайну спишется два банана, выставится два чека. И заметь — **никто не
падал**, код отработал идеально. Виновата сеть + слепой retry.

---

## Почему брокер не может сам догадаться

Записи выглядят одинаково:

```
orders: [ {товар: банан, кол-во: 1} , {товар: банан, кол-во: 1} ]
              ↑ первая                    ↑ повтор
```

На записи нет ничего, что говорит «это тот же самый заказ, что секунду
назад». Для брокера это просто две легитимные записи. Он их не различает.

---

## Как чинит идемпотентный producer

Включается одной настройкой (с Kafka 3.0 — **по умолчанию on**):

```java
props.put("enable.idempotence", "true");
```

Теперь на каждую запись producer вешает две бирки:

- **PID** — id этого producer'а (выдаёт брокер при старте).
- **Seq** — порядковый номер записи, растёт: 1, 2, 3, …

Брокер запоминает последний Seq, который он принял от этого PID, и
отбрасывает повтор:

```
checkout (PID=42)                 Broker (orders, помнит последний Seq)
   |                                |  последний seq: 0
   |  send(банан) Seq=1 ----------->|  1 > 0 → записал ✓, запомнил Seq=1
   |  <---- ack (ПОТЕРЯЛСЯ ✗) ------|      orders: [банан]
   |                                |
   |  retry: send(банан) Seq=1 ---->|  1 == 1 → УЖЕ ВИДЕЛ, выбросил повтор
   |  <-------- ack ----------------|      orders: [банан]   ← по-прежнему ОДИН
```

Повтор пришёл с тем же `Seq=1`. Брокер: «этот номер я уже принимал» —
молча выбрасывает, но всё равно отвечает `ack`, чтобы producer успокоился.
В логе **один** банан. Клиент кликнул раз — заказ один.

---

## Зачем PID, если retry делает тот же producer

Логичный вопрос: повтор шлёт тот же самый producer — зачем ему id? Затем,
что брокер обслуживает **не одного** producer'а. К нему одновременно
стучатся десятки приложений, и все шлют в один топик. Каждый нумерует свои
сообщения `Seq=1, 2, 3…` — со своей единицы.

Если бы бирка была только `Seq`, брокер видел бы кашу:

```
в топик orders валятся сообщения:
   Seq=1   Seq=1   Seq=2   Seq=1   Seq=2 ...
   чей это Seq=1?? повтор старого или первое сообщение нового producer'а?
```

Брокер не смог бы понять, чей `Seq=1`. `PID` подписывает каждую запись:
«это счётчик вот этого producer'а». Теперь всё однозначно:

```
   PID=42 Seq=1     ← producer A, первое           → записать
   PID=42 Seq=1     ← producer A, ПОВТОР            → тот же PID+Seq, выкинуть
   PID=99 Seq=1     ← producer B, первое            → другой PID, это НЕ повтор, записать
```

Брокер помнит последний Seq **отдельно для каждого PID**. `PID + Seq` — это
как «фамилия + номер листа»: в классе все ученики нумеруют свои листы
«1, 2, 3», и учитель не поймёт «лист 2» без фамилии. `PID` = фамилия,
`Seq` = номер листа. Один `Seq` без имени бессмысленен — у всех свои
единицы.

Короче: **PID нужен, чтобы твой Seq не спутался с чужим Seq.**

---

## На каком уровне живёт PID + Seq (и это НЕ то же, что offset или транзакция)

Счётчик Seq ведётся не на топик и не на сообщение целиком, а на пару
**(PID, партиция)**. У одного producer'а (`PID=42`), пишущего в топик из 3
партиций, — **три независимых счётчика**:

```
PID=42 → partition 0:  Seq 0,1,2,3...
PID=42 → partition 1:  Seq 0,1,2,3...   (свой отдельный счётчик)
PID=42 → partition 2:  Seq 0,1,2,3...   (свой отдельный)
```

Почему на партицию, а не на весь топик: партиции живут на **разных
брокерах**, и каждый брокер сам сторожит дубли для своих партиций. Общий
счётчик на топик потребовал бы координации между брокерами — дорого и
незачем.

Теперь разведём три уровня, чтобы они не слиплись в голове:

```
┌────────────────┬──────────────────────────────────────────────┐
│ offset         │ позиция записи В ЛОГЕ партиции. Ставит БРОКЕР  │
│                │ когда записал. Виден всем, постоянный. 0,1,2...│
├────────────────┼──────────────────────────────────────────────┤
│ Seq (+ PID)    │ внутренняя бирка producer'а ПРОТИВ ДУБЛЕЙ.     │
│                │ Ты её не видишь — только клиент и брокер.      │
│                │ Живёт на (PID, партиция). Это идемпотентность. │
├────────────────┼──────────────────────────────────────────────┤
│ транзакция     │ ещё выше: скобки begin…commit вокруг НЕСКОЛЬ-  │
│                │ КИХ записей (в разные топики/партиции), чтобы  │
│                │ применить их всё-или-ничего. Отдельный слой.   │
└────────────────┴──────────────────────────────────────────────┘
```

- `offset` — «какая ты по счёту в логе» (для читателей).
- `Seq + PID` — «не записал ли я тебя уже» (против дубля от retry).
- транзакция — «эти несколько записей вместе или никак».

Идемпотентность (PID+Seq) работает **сама по себе, без транзакций** —
включена по умолчанию. Транзакция — надстройка сверху: внутри она всё
равно опирается на PID+Seq, но добавляет «всё или ничего» поверх
нескольких записей. Так что PID+Seq — это **самый нижний слой**, не топик
и не транзакция.

---

## Граница: чего идемпотентность НЕ чинит

Это важно, иначе появится ложное ощущение «магии против всех дублей».

Идемпотентность ловит только повтор, который сделала **сама библиотека
Kafka внутри одной сессии** — как выше. Она НЕ поможет, если:

```
клиент случайно кликнул "Купить" ДВА раза
   → это два РАЗНЫХ заказа с точки зрения приложения
   → два send() с разными данными
   → идемпотентность их НЕ склеит (и правильно — вдруг он реально хотел два банана)

ИЛИ приложение упало и при рестарте само переслало заказ
   → новая сессия producer'а → новый PID
   → для брокера это новая запись → дубль пройдёт
```

Первый случай — вообще не про Kafka, это дедуп на уровне бизнес-логики
(идемпотентный ключ заказа). Второй случай — падение посреди
read-process-write — это то, что чинят **транзакции** (см. файл про банан
и транзакции).

---

## Два дубля — два лекарства, не путать

```
┌─────────────────────────────┬────────────────────────────────────┐
│  ДУБЛЬ ОТ RETRY              │  ДУБЛЬ ОТ ПАДЕНИЯ                    │
│  (сеть съела ack)           │  (упал между "записал" и "коммит")  │
├─────────────────────────────┼────────────────────────────────────┤
│  один клик → два банана     │  один заказ обработан дважды        │
│  никто не падал             │  сервис упал и перечитал            │
│  ЛЕЧИТ: идемпотентность     │  ЛЕЧИТ: транзакции                  │
│  (PID + Seq)                │  (begin…commit + offset внутри)     │
└─────────────────────────────┴────────────────────────────────────┘
```

Идемпотентность — дешёвая, включена по умолчанию, закрывает первый
столбец бесплатно. Транзакции — дороже, включаешь сознательно, закрывают
второй. Вместе они дают exactly-once для «прочитал → обработал → записал».

---

> Связь с кодом лабы: `Ex1Problem` (идемпотентность выключена) и
> `Ex2Idempotent` (включена) — это ровно левый столбец. Чтобы увидеть
> дубль вживую, в лабе рвётся соединение (`ss -K`), имитируя потерянный
> `ack`. С выключенной идемпотентностью появляется дубль-банан, с
> включённой — нет.
